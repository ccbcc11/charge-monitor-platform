# 04-接口文档说明

## 1. 文档信息

| 项目       | 内容                                                         |
| ---------- | ------------------------------------------------------------ |
| 项目名称   | 新能源充电设施运行监测与智能告警平台                         |
| 后端项目名 | charge-monitor-backend                                       |
| 数据库名   | charge_monitor                                               |
| 文档版本   | v1.0 / MVP 第一版                                            |
| 适用阶段   | 第一阶段：MVP 开发                                           |
| 编写目的   | 明确 MVP 第一版接口路径、请求参数、响应结构、状态码和开发顺序 |

------

## 2. 接口设计原则

1. 接口统一以 `/api` 作为前缀。
2. 接口返回统一使用 `Result<T>`。
3. 分页接口统一使用 `pageNo`、`pageSize`。
4. 时间格式统一为 `yyyy-MM-dd HH:mm:ss`。
5. 受保护接口必须携带 `satoken` 请求头。
6. 新增和修改接口使用 DTO 接收参数。
7. 查询结果使用 VO 返回，避免直接返回数据库 Entity。
8. 删除设备采用逻辑删除，不物理删除。
9. MVP 阶段告警规则写死在代码中，暂不开放规则配置接口。
10. 第二阶段再扩展 RabbitMQ、WebSocket、工单、操作日志、日报接口。

## 4. 统一返回结构

### 4.1 成功返回

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {},
  "timestamp": "2026-06-26 14:30:00"
}
```

### 4.2 失败返回

```json
{
  "code": 50001,
  "message": "业务处理失败",
  "data": null,
  "timestamp": "2026-06-26 14:30:00"
}
```

### 4.3 分页返回

分页接口的 `data` 统一采用以下结构：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "records": [],
    "total": 100,
    "pageNo": 1,
    "pageSize": 10,
    "pages": 10
  },
  "timestamp": "2026-06-26 14:30:00"
}
```

### 4.4 通用状态码

| code  | 含义                   | 说明                         |
| ----- | ---------------------- | ---------------------------- |
| 200   | 操作成功               | 请求正常处理                 |
| 400   | 请求参数错误           | 参数为空、格式错误、类型错误 |
| 401   | 用户未登录或登录已过期 | token 缺失或失效             |
| 403   | 无权限访问             | 当前用户无操作权限           |
| 404   | 请求资源不存在         | 查询的数据不存在             |
| 405   | 请求方法不支持         | GET/POST/PUT/DELETE 使用错误 |
| 500   | 系统异常               | 未预期的系统错误             |
| 50001 | 业务处理失败           | 通用业务异常                 |
| 50002 | 数据不存在             | 查询对象不存在               |
| 50003 | 数据已存在             | 唯一字段重复                 |
| 51001 | 设备不存在             | deviceCode 或 id 无效        |
| 51002 | 设备已停用             | 设备 runningStatus 为停用    |
| 52002 | 告警不存在             | 告警记录不存在               |

# 6. 认证模块接口

## 6.1 用户登录

### 基本信息

| 项目         | 内容              |
| ------------ | ----------------- |
| 接口名称     | 用户登录          |
| 请求方式     | POST              |
| 接口路径     | `/api/auth/login` |
| 是否需要登录 | 否                |
| 请求 DTO     | `LoginDTO`        |
| 响应 VO      | `LoginVO`         |

### 请求参数

| 字段     | 类型   | 是否必填 | 说明   | 示例   |
| -------- | ------ | -------- | ------ | ------ |
| username | string | 是       | 用户名 | admin  |
| password | string | 是       | 密码   | 123456 |

### 请求示例

```json
{
  "username": "admin",
  "password": "123456"
}
```

### 响应字段

| 字段      | 类型   | 说明             |
| --------- | ------ | ---------------- |
| token     | string | 登录 token       |
| tokenName | string | token 请求头名称 |
| userId    | number | 用户 ID          |
| username  | string | 用户名           |
| realName  | string | 真实姓名         |
| roles     | array  | 角色编码列表     |

### 响应示例

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "token": "8f72c7d0-76f9-4f5d-9a1d-2b7c16a0e111",
    "tokenName": "satoken",
    "userId": 1,
    "username": "admin",
    "realName": "系统管理员",
    "roles": ["admin"]
  },
  "timestamp": "2026-06-26 14:30:00"
}
```

### 处理逻辑

```text
1. 校验 username、password 是否为空
2. 根据 username 查询 sys_user
3. 判断用户是否存在
4. 判断用户状态是否启用
5. 校验密码
6. 查询用户角色
7. 调用 Sa-Token 登录
8. 返回 token 和用户基础信息
```

```text

4. 若设备停用，返回设备已停用
```

| CONTINUOUS  | 连续异常告警，第二阶段完善 |
| FLUCTUATION | 波动异常告警，第二阶段完善 |

## 14.5 告警指标 alarmMetric

| 值            | 含义     |
| ------------- | -------- |
| temperature   | 温度     |
| voltage       | 电压     |
| current       | 电流     |
| power         | 功率     |
| network_delay | 网络延迟 |

## 14.6 告警等级 alarmLevel

| 值   | 含义 |
| ---- | ---- |
| 1    | 一般 |
| 2    | 重要 |
| 3    | 严重 |

## 14.7 告警状态 alarmStatus

| 值   | 含义                           |
| ---- | ------------------------------ |
| 0    | 未确认                         |
| 1    | 已确认                         |
| 2    | 已恢复                         |
| 3    | 已关闭，第二阶段工单闭环时使用 |

# 15. Redis Key 约定

| Key                             | 类型          | 说明                             |
| ------------------------------- | ------------- | -------------------------------- |
| `device:status:{deviceCode}`    | String / JSON | 设备最新运行状态                 |
| `device:heartbeat:{deviceCode}` | String        | 设备最近心跳时间                 |
| `device:online:set`             | Set           | 当前在线设备编号集合             |
| `device:alarm:set`              | Set           | 当前存在未恢复告警的设备编号集合 |

示例：

```text
device:status:CP-0001
device:heartbeat:CP-0001
device:online:set
device:alarm:set
```

# 16. 告警去重规则

## 16.1 去重条件

同一未恢复告警由以下条件确定：

```text
deviceCode + alarmType + alarmMetric + alarmStatus in (0, 1)
```

## 16.2 新增告警

如果不存在未恢复告警：

```text
新增 alarm_record
alarm_status = 0
alarm_count = 1
first_time = 当前时间
last_time = 当前时间
```

## 16.3 更新告警

如果已存在未恢复告警：

```text
不新增记录
更新 current_value
更新 last_time
alarm_count = alarm_count + 1
更新 alarm_message
```
