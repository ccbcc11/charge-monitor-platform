## 3. 测试准备

### 3.1 启动服务

```bash
cd charge-monitor-backend
mvn spring-boot:run
```

### 3.2 确认服务正常

浏览器访问 http://localhost:8080/doc.html，能打开 Knife4j 页面即表示启动成功。

### 3.3 测试 JSON 文件

所有测试 JSON 位于 `docs/test-json/` 目录：

```
test-json/
├── login.json                    # 登录
├── create-device.json            # 新增设备
├── update-device.json            # 修改设备
├── data-report-normal.json       # 上报正常数据
├── data-report-temp-alarm.json   # 上报触发温度告警
├── data-report-volt-alarm.json   # 上报触发电压告警
└── data-report-delay-alarm.json  # 上报触发网络延迟告警
```

## 4. 测试流程（按顺序）

### 步骤 1：登录获取 Token

**请求：**
```bash
curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"123456"}'
```

或使用 `docs/test-json/login.json`

**预期响应：**
```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "token": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "tokenName": "satoken",
    "userId": 1,
    "username": "admin",
    "realName": "系统管理员",
    "roles": ["admin"]
  }
}
```

**⚠️ 重要：** 保存返回的 `token` 值，后续所有接口都需要在请求头中携带：
```
satoken: <你的token>
```

### 步骤 2：测试鉴权拦截

不携带 token 直接访问受保护接口：
```bash
curl -X GET http://localhost:8080/api/device/page
```

**预期响应：** 返回 401，`"message": "未登录或会话已失效"`

---

### 步骤 3：设备管理

#### 3.1 分页查询设备

```bash
curl -X GET "http://localhost:8080/api/device/page?pageNo=1&pageSize=10" \
  -H "satoken: <你的token>"
```

**验证点：** 返回 5 条设备记录（init.sql 中预置的 5 台设备）。

#### 3.2 按条件筛选

```bash
curl -X GET "http://localhost:8080/api/device/page?region=城东区&onlineStatus=1" \
  -H "satoken: <你的token>"
```

**验证点：** 只返回城东区在线设备。

#### 3.3 新增设备

```bash
curl -X POST http://localhost:8080/api/device \
  -H "Content-Type: application/json" \
  -H "satoken: <你的token>" \
  -d @docs/test-json/create-device.json
```

**验证点：** 返回新设备 ID（应该是 6）。

#### 3.4 查询设备详情

```bash
curl -X GET http://localhost:8080/api/device/1 \
  -H "satoken: <你的token>"
```

**验证点：** 返回 CP-0001 的完整信息。

#### 3.5 修改设备

```bash
curl -X PUT http://localhost:8080/api/device/1 \
  -H "Content-Type: application/json" \
  -H "satoken: <你的token>" \
  -d @docs/test-json/update-device.json
```

**验证点：** `code: 200`。

#### 3.6 删除设备

```bash
curl -X DELETE http://localhost:8080/api/device/6 \
  -H "satoken: <你的token>"
```

**验证点：** `code: 200`。再次分页查询时该设备不再出现，但数据库中 `deleted=1`。

---

### 步骤 4：设备数据上报（核心测试）

#### 4.1 上报正常数据

```bash
curl -X POST http://localhost:8080/api/device/data/report \
  -H "Content-Type: application/json" \
  -H "satoken: <你的token>" \
  -d @docs/test-json/data-report-normal.json
```

**验证点：**
```json
{
  "code": 200,
  "data": {
    "dataId": <自动生成>,
    "deviceCode": "CP-0001",
    "alarmTriggered": false,
    "alarmIds": []
  }
}
```

#### 4.2 查询设备最新状态

```bash
curl -X GET http://localhost:8080/api/device/data/latest/CP-0001 \
  -H "satoken: <你的token>"
```

**验证点：**
- `fromCache: true`（说明从 Redis 读取）
- 各项指标与上报数据一致
- `onlineStatus: 1`（设备已标记在线）

#### 4.3 查询历史数据

```bash
curl -X GET "http://localhost:8080/api/device/data/history?deviceCode=CP-0001&pageNo=1&pageSize=10" \
  -H "satoken: <你的token>"
```

**验证点：** 返回 CP-0001 的历史上报记录。

---

### 步骤 5：告警触发测试

#### 5.1 触发温度告警（严重）

```bash
curl -X POST http://localhost:8080/api/device/data/report \
  -H "Content-Type: application/json" \
  -H "satoken: <你的token>" \
  -d @docs/test-json/data-report-temp-alarm.json
```

**验证点：**
```json
{
  "code": 200,
  "data": {
    "alarmTriggered": true,
    "alarmIds": [<告警ID>]
  }
}
```

#### 5.2 验证告警去重

再次执行同一条温度告警上报。

**验证点：**
- `alarmTriggered: true`
- 告警 ID 与上次相同（同一告警被更新，而非新建）
- 查询该告警详情，`alarmCount` 变为 2

#### 5.3 触发电压告警（重要）

```bash
curl -X POST http://localhost:8080/api/device/data/report \
  -H "Content-Type: application/json" \
  -H "satoken: <你的token>" \
  -d @docs/test-json/data-report-volt-alarm.json
```

**验证点：** `alarmTriggered: true`，生成电压告警。

#### 5.4 触发网络延迟告警（一般）

```bash
curl -X POST http://localhost:8080/api/device/data/report \
  -H "Content-Type: application/json" \
  -H "satoken: <你的token>" \
  -d @docs/test-json/data-report-delay-alarm.json
```

**验证点：** `alarmTriggered: true`，生成网络延迟告警。

---

### 步骤 6：告警处理

#### 6.1 分页查询告警

```bash
curl -X GET "http://localhost:8080/api/alarm/record/page?pageNo=1&pageSize=10&alarmStatus=0" \
  -H "satoken: <你的token>"
```

**验证点：** 返回未确认的告警列表。

#### 6.2 查询告警详情

```bash
curl -X GET http://localhost:8080/api/alarm/record/<告警ID> \
  -H "satoken: <你的token>"
```

**验证点：** 返回告警完整信息，包括设备名称、站点、区域。

#### 6.3 确认告警

```bash
curl -X PUT http://localhost:8080/api/alarm/record/<告警ID>/ack \
  -H "satoken: <你的token>"
```

**验证点：** `code: 200`。再次查询告警详情，`alarmStatus` 变为 1。

#### 6.4 恢复告警

```bash
curl -X PUT http://localhost:8080/api/alarm/record/<告警ID>/recover \
  -H "satoken: <你的token>"
```

**验证点：** `code: 200`。再次查询告警详情，`alarmStatus` 变为 2，`recoverTime` 有值。

---

### 步骤 7：报表验证

```bash
curl -X GET http://localhost:8080/api/report/overview \
  -H "satoken: <你的token>"
```

**验证点：**
- `deviceTotal`: 5（5 台设备，删除后变 4）
- `onlineCount`: 上报过数据的设备数
- `offlineCount`: `deviceTotal - onlineCount`
- `todayAlarmCount`: 今天已生成的告警数
- `onlineRate`: 在线率百分比

## 5. Redis 验证

测试过程中可以通过 Redis 客户端验证缓存数据：

```bash
# 连接 Redis
redis-cli

# 查看设备最新状态
GET device:status:CP-0001

# 查看设备心跳

# 查看在线设备集合
SMEMBERS device:online:set

# 查看告警设备集合
SMEMBERS device:alarm:set
```

## 6. 数据库验证

```sql
-- 查看设备列表
SELECT id, device_code, device_name, online_status, running_status, last_heartbeat
FROM device_info WHERE deleted = 0;

-- 查看今天上报的运行数据
SELECT * FROM device_data WHERE DATE(create_time) = CURDATE() ORDER BY id DESC;

-- 查看今天产生的告警
SELECT id, alarm_no, device_code, alarm_type, alarm_metric, alarm_level, alarm_status, alarm_count
FROM alarm_record WHERE DATE(create_time) = CURDATE() ORDER BY id DESC;
```

## 7. 异常场景测试

| 场景 | 请求 | 预期结果 |
|---|---|---|
| 设备不存在 | 上报数据时使用不存在的 deviceCode | 返回 51001 设备不存在 |
| 设备已停用 | 将设备 runningStatus 改为 0 后上报 | 返回 51002 设备已停用 |
| 告警已恢复后确认 | 对已恢复告警执行 ack | 返回错误 "告警已恢复，不能确认" |
| 参数为空 | 登录时不传 username | 返回 400 参数错误 |
| Token 过期 | 等待 24 小时后访问接口 | 返回 401 未登录 |
