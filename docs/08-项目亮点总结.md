# 08-项目亮点总结

> 项目名称：新能源充电设施运行监测与智能告警平台  
> 文档定位：项目复盘文档 + 技术亮点说明书 + 面试备考资料  
> 适用场景：项目答辩、简历包装、面试复习、后续功能扩展参考

---

## 1. 项目整体定位

本项目面向新能源充电设施运维场景，围绕“设备运行数据采集、状态监测、异常告警、实时通知、工单处置”构建一套后端平台。

项目不是单纯 CRUD，而是围绕设备运行数据形成了一条完整业务链路：

```text
设备上报运行数据
→ 后端保存运行数据
→ 更新设备在线状态和 Redis 最新状态
→ RabbitMQ 异步投递告警检测任务
→ 告警消费者读取动态规则
→ Redis 缓存启用规则降低数据库压力
→ 阈值告警 / 连续异常告警检测
→ 生成或更新 alarm_record
→ 严重告警自动生成 work_order
→ WebSocket 实时推送给前端
→ 运维人员接单、处理、完成或关闭工单
```

从业务角度看，项目完成了：

```text
数据采集 → 异常识别 → 实时通知 → 自动派单 → 人工处置
```

从技术角度看，项目综合使用了 Spring Boot、MyBatis-Plus、MySQL、Redis、RabbitMQ、WebSocket、Sa-Token、Knife4j、Spring Task 与 Spring 事务 afterCommit 回调。

## 2. 完整业务闭环

### 2.1 设备运行监测链路

设备通过接口上报运行数据，包括电压、电流、功率、温度、SOC、网络延迟、故障码、上报时间等。

后端接收后完成：

1. 校验设备是否存在；
2. 校验设备是否停用；
3. 保存运行数据到 `device_data`；
4. 更新设备在线状态和最近心跳到 `device_info`；
5. 写入 Redis 最新状态；
6. 写入 Redis 在线设备集合；
7. 事务提交成功后投递 RabbitMQ 消息。

### 2.2 异步告警检测链路

RabbitMQ 消费者消费设备数据消息，根据 `deviceDataId` 查询运行数据，再调用告警检测服务。

当前支持三类告警：

1. `THRESHOLD`：单次阈值告警；
2. `CONTINUOUS`：连续异常告警；
3. `OFFLINE`：设备离线告警。

其中：

- `THRESHOLD` 和 `CONTINUOUS` 由 RabbitMQ 消费者异步处理；
- `OFFLINE` 由 Spring Task 定时检测设备心跳实现。

### 2.3 告警通知链路

告警生成或更新后，后端通过 WebSocket 主动推送给浏览器客户端。

前端无需轮询告警列表，可以实时收到告警编号、设备编号、告警类型、告警指标、告警等级、当前值、阈值、告警内容、首次告警时间、最近告警时间和推送时间。

### 2.4 工单处置链路

严重告警自动生成运维工单。

运维人员可以分页查询工单、查看工单详情、接单、完成工单、关闭工单、删除工单。

工单状态流转：

```text
待处理 0
  ↓ 接单
处理中 1
  ↓ 完成
已完成 2

待处理 0 / 处理中 1
  ↓ 关闭
已关闭 3
```

这样项目形成了完整运维闭环：

```text
设备异常 → 告警记录 → 实时推送 → 自动工单 → 人工处理
```

## 4. 亮点一：RabbitMQ 异步告警检测

### 4.1 为什么要做？

设备数据上报接口原本可能同时承担这些任务：校验设备、保存运行数据、更新设备在线状态、更新 Redis、查询告警规则、执行告警检测、生成告警记录、推送通知、自动生成工单。

如果全部放在 HTTP 请求线程中同步执行，接口会越来越重。当设备数量增加、上报频率提高、规则数量变多时，会出现接口响应变慢、设备上报被阻塞、告警检测失败影响数据入库、后续扩展困难等问题。

因此项目引入 RabbitMQ，将“设备数据上报”和“告警检测”解耦。

### 4.2 不这么做会有什么问题？

1. 设备上报接口耗时变长；
2. 告警规则查询压力直接落在上报接口上；
3. 告警入库失败可能导致整个上报事务失败；
4. WebSocket 推送、自动工单逻辑会继续堆在接口里；
5. 高并发情况下不利于削峰；
6. 后续拆分服务时成本更高。

### 4.3 项目中如何实现？

项目设计了设备数据上报消息 `DeviceDataReportMessage`，核心字段包括 `deviceDataId`、`deviceId`、`deviceCode`、`reportTime`。

设备上报接口完成核心数据入库后，在事务提交成功后发送 MQ 消息：

```text
exchange: charge.monitor.device.data.alarm.exchange
queue:    charge.monitor.device.data.alarm.queue
routing:  charge.monitor.device.data.report
```

消费者 `DeviceDataAlarmConsumer` 收到消息后：

1. 根据 `deviceDataId` 查询 `device_data`；
2. 根据 `deviceId` 或 `deviceCode` 查询 `device_info`；
3. 调用 `AlarmDetectService` 执行告警检测；
4. 检测完成后执行 WebSocket 推送和自动工单生成。

### 4.4 带来的收益

1. HTTP 上报接口变轻；
2. 告警检测与设备上报解耦；
3. RabbitMQ 可以削峰填谷；
4. 消费失败可以重试；
5. 后续可以扩展多个消费者；
6. WebSocket 推送和自动工单可以作为后置动作接入；
7. 项目结构更接近真实生产系统。

### 4.5 涉及技术点

RabbitMQ Exchange、Queue、RoutingKey、Spring AMQP、RabbitTemplate、`@RabbitListener`、JSON 消息序列化、消费重试、事务提交后投递 MQ、消费幂等。

### 4.6 面试常问问题

#### Q1：为什么不用线程池异步，而要用 RabbitMQ？

线程池只能解决当前服务内部的异步执行问题，但不具备可靠消息、消息持久化、消费确认、失败重试、削峰填谷等能力。

如果服务重启，线程池中的任务可能丢失；而 RabbitMQ 可以将消息持久化到队列中，消费者恢复后继续消费。

所以对于设备数据上报与告警检测这种需要解耦、削峰、可靠处理的场景，RabbitMQ 更合适。

#### Q2：为什么设备数据入库后不直接发送 MQ，而要 afterCommit？

因为事务方法执行到最后一行时，数据库事务不一定已经真正提交。

如果此时发送 MQ，消费者可能先消费消息，并根据 `deviceDataId` 查询数据，但数据还没提交，消费者可能查不到。

因此项目中使用 `TransactionSynchronizationManager` 注册 `afterCommit` 回调，确保数据库提交成功后再发送 MQ。

#### Q3：MQ 消息重复消费怎么办？

项目通过幂等设计控制重复消费影响：告警记录按照设备、告警类型、告警指标、未恢复状态去重；如果已有未恢复告警，则更新 `alarm_count` 和 `last_time`，不重复新增；工单表对 `alarm_id` 建唯一索引，避免重复派单。

#### Q4：MQ 消费失败怎么办？

当前项目配置了消费者重试，消费者抛异常时会触发重试。

对于更完整的生产方案，可以继续扩展手动 ACK、死信队列 DLQ、失败消息记录表、告警补偿任务、消费失败告警通知。

## 5. 亮点二：动态告警规则 `alarm_rule`

### 5.1 为什么要做？

如果阈值直接写死在代码中，例如 `temperature > 80`、`voltage < 180`、`networkDelay > 200`，那么每次规则调整都要改代码、重新打包、重新部署。

这不符合运维平台的要求。告警规则应该可以配置化。因此项目设计了 `alarm_rule` 表，用于存储告警规则。

### 5.2 不这么做会有什么问题？

1. 修改阈值需要改代码；
2. 运维人员无法通过后台管理规则；
3. 不同设备类型无法配置不同规则；
4. 规则启停不方便；
5. 后续扩展连续异常、波动异常困难；
6. 项目不像真实后台系统。

### 5.3 项目中如何实现？

`alarm_rule` 表核心字段包括规则编码、规则名称、告警类型、适用设备类型、指标名称、运算符、阈值、窗口大小、触发次数、告警等级、是否启用。

当前支持规则类型：

```text
THRESHOLD   单次阈值告警
CONTINUOUS  连续异常告警
OFFLINE     离线告警
FLUCTUATION 波动异常，预留
```

并实现了告警规则管理接口：

```text
POST   /api/alarm/rule
PUT    /api/alarm/rule/{id}
DELETE /api/alarm/rule/{id}
GET    /api/alarm/rule/page
GET    /api/alarm/rule/{id}
PUT    /api/alarm/rule/{id}/enable
PUT    /api/alarm/rule/{id}/disable
```

### 5.4 带来的收益

1. 告警阈值不再硬编码；
2. 支持后台动态新增、修改、启停规则；
3. 支持不同类型告警扩展；
4. 支持规则缓存；
5. 提高系统可配置性；
6. 更接近真实运维平台。

### 5.5 涉及技术点

规则表设计、DTO / VO / Entity 分层、MyBatis-Plus、逻辑删除、唯一索引、Jakarta Validation、Knife4j 接口文档、规则合法性校验。

### 5.6 面试常问问题

#### Q1：为什么要设计 DTO、VO，而不是直接用 Entity？

Entity 对应数据库表结构，不适合直接暴露给前端。

DTO 用于接收请求参数，方便做参数校验；VO 用于返回前端，控制返回字段和展示结构。

这样可以避免数据库结构和接口结构强耦合，也更符合分层设计。

#### Q2：为什么规则编码 `rule_code` 要唯一？

规则编码是规则的业务唯一标识，用于缓存、日志、连续异常窗口 key 等场景。

如果重复，会导致规则识别混乱。因此数据库对 `rule_code` 建唯一索引。

#### Q3：为什么修改规则后要清 Redis 缓存？

因为消费者检测告警时优先读取 Redis 中的启用规则。

如果数据库规则修改后不清缓存，系统可能继续使用旧规则。项目采用 Cache Aside 模式，规则修改后删除缓存，下一次检测重新加载数据库最新规则。

## 6. 亮点三：Redis 告警规则缓存

### 6.1 为什么要做？

RabbitMQ 消费者每处理一条设备数据，都需要读取启用的告警规则。

如果每条数据都查询 MySQL 的 `alarm_rule` 表，在设备数量增加、上报频率提高后，会造成大量重复查询。告警规则属于“读多写少”的数据，非常适合缓存。

### 6.2 项目中如何实现？

项目新增 `AlarmRuleCacheService`。

读取规则流程：

```text
1. 根据 alarmType 拼接 Redis key；
2. 优先读取 Redis；
3. Redis 命中则反序列化为规则列表；
4. Redis 未命中则查询 MySQL；
5. 将查询结果写入 Redis；
6. 返回规则列表。
```

缓存 key 示例：

```text
alarm:rule:enabled:THRESHOLD
alarm:rule:enabled:CONTINUOUS
```

规则变更流程：

```text
新增 / 修改 / 删除 / 启用 / 禁用规则
→ 更新 MySQL
→ 删除 Redis 规则缓存
→ 下一次告警检测自动重建缓存
```

### 6.3 为什么采用 Cache Aside？

Cache Aside 模式：

```text
读：先查缓存，缓存没有查数据库，再写缓存
写：先更新数据库，再删除缓存
```

项目选择这种方式，是因为规则属于读多写少数据，Cache Aside 简单、稳定、容易理解。

### 6.4 为什么删除缓存，而不是更新缓存？

删除缓存比更新缓存更简单，也更不容易出现并发不一致。

如果每次修改规则后主动更新缓存，需要重新查询所有启用规则并写入 Redis。并发场景下可能出现旧数据覆盖新数据的问题。

删除缓存后，下一次读取时重新加载数据库最新数据，更稳。

### 6.5 带来的收益

1. 降低 `alarm_rule` 表查询压力；
2. 提高告警消费者处理速度；
3. 支持规则动态变更；
4. 通过缓存空数组避免缓存穿透；
5. Redis 故障时可以回退查询 MySQL。

### 6.6 面试常问问题

#### Q1：缓存和数据库如何保持一致？

项目采用 Cache Aside 模式。读取时优先查 Redis，未命中查 MySQL 并回填 Redis。写入时先修改 MySQL，再删除 Redis 缓存。这样规则变更后，下一次告警检测会重新加载数据库最新规则，避免长期使用旧缓存。

#### Q2：Redis 挂了怎么办？

Redis 是缓存，不是规则的唯一数据源。如果读取 Redis 失败，项目会回退查询 MySQL，保证告警规则仍然可以正常加载。Redis 恢复后，下一次可以重新写入缓存。

#### Q3：为什么空规则列表也要缓存？

如果数据库中没有启用规则，而不缓存空结果，那么每条设备数据都会查询数据库，形成缓存穿透。缓存空数组 `[]` 可以避免无效请求反复打到数据库。

#### Q4：缓存过期时间为什么还需要？

即使规则管理接口忘记清缓存，TTL 到期后缓存也会自动失效，系统会重新加载数据库规则。TTL 是一种兜底机制。

## 7. 亮点四：Redis List 连续异常滑动窗口

### 7.1 为什么要做？

单次阈值告警只判断当前一次数据是否异常。

但真实运维场景中，有些异常需要观察一段时间。比如温度偶尔超过 75℃ 可能是瞬时波动，但最近 5 次里有 3 次超过 75℃ 才说明设备持续偏热。

因此项目设计了 `CONTINUOUS` 连续异常告警。

### 7.2 规则含义

例如规则：

```text
RULE_TEMP_CONTINUE
metric_name = temperature
operator = >
threshold_value = 75
window_size = 5
trigger_count = 3
```

含义：

```text
最近 5 次上报中，有 3 次 temperature > 75℃，则触发连续高温告警。
```

### 7.3 项目中如何实现？

项目使用 Redis List 维护每个设备、每条规则的最近 N 次判断结果。

Key 示例：

```text
alarm:continuous:window:CP-0001:RULE_TEMP_CONTINUE
```

每次设备数据到达时：

1. 读取启用的 `CONTINUOUS` 规则；
2. 判断当前值是否命中规则；
3. 命中写入 `1`；
4. 未命中写入 `0`；
5. 使用 `LPUSH` 写入最新结果；
6. 使用 `LTRIM` 只保留最近 `window_size` 条；
7. 使用 `LRANGE` 读取窗口；
8. 统计 `1` 的数量；
9. 如果异常次数大于等于 `trigger_count`，生成连续异常告警。

### 7.4 为什么用 Redis List？

Redis List 天然适合维护滑动窗口：

```text
LPUSH  写入最新结果
LTRIM  保留最近 N 条
LRANGE 读取最近 N 条
EXPIRE 设置窗口过期时间
```

相比 MySQL 查询最近 N 条数据，Redis List 更轻量，更适合高频状态判断。

### 7.5 带来的收益

1. 支持状态型异常检测；
2. 告警不再只依赖单条数据；
3. 能过滤部分瞬时波动；
4. `window_size` 和 `trigger_count` 可配置；
5. Redis 操作简单高效；
6. 丰富了项目技术亮点。

### 7.6 面试常问问题

#### Q1：为什么不用 MySQL 查询最近 N 条数据？

MySQL 查询最近 N 条设备数据通常需要按时间排序和扫描。设备上报频率高时，频繁查询会增加数据库压力。Redis List 更适合维护短窗口状态，每次只保留最近 N 次判断结果，读写成本更低。

#### Q2：Redis List 会不会无限增长？

不会。每次写入后都会执行 `LTRIM`，只保留最近 `window_size` 条数据。同时设置过期时间，避免长期不上报的设备窗口占用内存。

#### Q3：连续异常和阈值异常有什么区别？

阈值异常只看当前一条数据，适合单次越界立即告警。连续异常关注最近一段时间内的异常频率，适合识别持续性问题。

#### Q4：连续异常告警会不会重复生成？

不会无限新增。项目仍然使用告警去重逻辑：同一设备、同一告警类型、同一指标、未恢复状态下只保留一条告警，后续触发只更新 `alarm_count` 和 `last_time`。

## 8. 亮点五：WebSocket 实时告警推送

### 8.1 为什么要做？

告警系统强调实时性。

如果前端只能通过轮询查询告警列表，会存在延迟，也会产生大量无效请求。WebSocket 可以在告警生成后由后端主动推送给前端，更适合告警、监控大屏、实时通知等场景。

### 8.2 项目中如何实现？

项目新增 WebSocket 连接地址：

```text
ws://localhost:8080/ws/alarm
```

核心类：

```text
WebSocketConfig
AlarmWebSocketHandler
AlarmPushService
AlarmPushServiceImpl
AlarmPushMessage
```

实现流程：

1. 前端连接 `/ws/alarm`；
2. `AlarmWebSocketHandler` 保存在线 WebSocket session；
3. RabbitMQ 消费者完成告警检测；
4. 告警事务提交成功后调用 `AlarmPushService`；
5. 查询 `alarm_record`；
6. 转换为 `AlarmPushMessage`；
7. JSON 序列化；
8. 广播给所有在线客户端。

### 8.3 为什么要事务提交后推送？

如果告警记录还没提交就推送，前端收到消息后立刻查询详情，可能查不到数据。因此项目使用 `afterCommit`，保证 `alarm_record` 提交成功后再推送。

### 8.4 推送失败怎么办？

WebSocket 是通知能力，不是核心事务。告警入库成功才是核心业务。项目中推送失败会捕获异常、记录日志、移除异常 session，但不会回滚告警事务，也不会影响 MQ 消费成功。

### 8.5 带来的收益

1. 告警实时到达前端；
2. 减少前端轮询；
3. 更适合监控大屏和运维平台；
4. 告警链路更完整；
5. 提升项目展示效果。

### 8.6 面试常问问题

#### Q1：WebSocket 和 HTTP 轮询有什么区别？

HTTP 轮询是前端定时请求后端，存在延迟和无效请求。WebSocket 是客户端和服务端建立长连接，服务端可以主动推送消息，更适合实时告警。

#### Q2：为什么第一版采用广播，而不是按用户推送？

MVP 阶段主要验证实时推送链路，广播模式实现简单可靠。后续可以结合用户角色、站点权限、设备负责人，实现定向推送。

#### Q3：WebSocket 推送失败会不会影响告警生成？

不会。告警入库是核心业务，WebSocket 只是后置通知。项目中推送异常会捕获并记录日志，不影响告警事务。

#### Q4：多个线程同时推送会不会有问题？

可能会。`WebSocketSession` 不建议多个线程同时调用 `sendMessage`，项目中可以对 session 加锁，避免并发发送异常。

## 9. 亮点六：严重告警自动生成工单

### 9.1 为什么要做？

告警只是发现问题，工单才是处理问题的入口。

如果系统只能生成告警，但没有后续处置流程，运维人员仍然需要人工记录和分配任务，业务闭环不完整。因此项目设计严重告警自动生成工单机制。

### 9.2 项目中如何实现？

当 RabbitMQ 消费者完成告警检测后，拿到 `alarmIds`。

在告警事务提交成功后，调用：

```text
WorkOrderAutoGenerateService.generateForAlarmIds(alarmIds)
```

自动生成条件：

```text
alarm_level = 3
alarm_status in (0, 1)
work_order_generated = 0
work_order 表中不存在相同 alarm_id 的工单
```

满足条件后：

1. 创建 `work_order`；
2. 设置工单状态为待处理；
3. 设置优先级为高；
4. 回写 `alarm_record.work_order_generated = 1`。

### 9.3 如何防止重复派单？

项目采用双保险：

1. `alarm_record.work_order_generated` 业务标记；
2. `work_order.alarm_id` 唯一索引。

即使 MQ 重复消费或并发执行，数据库唯一索引也可以兜底避免一条告警生成多个工单。

### 9.4 工单状态流转

工单状态：

```text
0：待处理
1：处理中
2：已完成
3：已关闭
```

允许流转：

```text
待处理 → 处理中 → 已完成
待处理 / 处理中 → 已关闭
```

不允许：

```text
已完成 → 处理中
已关闭 → 处理中
已关闭 → 已完成
```

### 9.5 带来的收益

1. 严重告警自动进入处置流程；
2. 减少人工派单；
3. 避免严重问题遗漏；
4. 告警到工单形成闭环；
5. 增强项目业务完整性。

### 9.6 面试常问问题

#### Q1：为什么只给严重告警生成工单？

如果所有告警都自动生成工单，会造成工单过多，影响运维效率。第一版只针对 `alarm_level = 3` 的严重告警生成工单，保证工单代表真正需要处理的问题。

#### Q2：如何保证同一个告警不会重复生成工单？

项目中使用 `work_order_generated` 字段做业务标记，同时在 `work_order` 表对 `alarm_id` 建唯一索引。业务判断和数据库约束双重保证幂等。

#### Q3：工单完成后是否自动恢复告警？

第一版没有自动恢复。因为工单完成代表处理动作完成，但不一定代表设备状态已经恢复。告警恢复仍然通过告警恢复接口处理。后续可以扩展为完成工单时可选同步恢复告警。

#### Q4：为什么自动工单失败不影响 MQ 消费？

因为告警记录是核心业务，自动工单属于后置动作。如果自动工单失败导致 MQ 反复重试，可能造成告警重复更新和推送混乱。因此项目中自动工单异常会记录日志，后续可通过补偿任务处理。

## 10. 亮点七：事务 afterCommit 后置动作设计

### 10.1 为什么要做？

项目中有多个动作依赖数据库事务成功提交：

1. 设备数据入库后发送 MQ；
2. 告警记录入库后 WebSocket 推送；
3. 告警记录入库后自动生成工单。

如果这些动作在事务提交前执行，可能出现数据不一致。

### 10.2 项目中哪些地方用了 afterCommit？

#### 1. 设备上报后发送 MQ

```text
device_data 事务提交成功
→ afterCommit
→ 发送 RabbitMQ 消息
```

避免消费者查不到刚插入的数据。

#### 2. 告警生成后 WebSocket 推送

```text
alarm_record 事务提交成功
→ afterCommit
→ 推送 WebSocket 消息
```

避免前端收到推送后查不到告警详情。

#### 3. 告警生成后自动工单

```text
alarm_record 事务提交成功
→ afterCommit
→ 创建 work_order
```

避免工单关联未提交或已回滚的告警。

### 10.3 带来的收益

1. 避免消息先于数据提交；
2. 避免前端收到不存在的告警；
3. 避免工单关联脏告警；
4. 提高数据一致性；
5. 让后置动作与核心事务解耦。

### 10.4 面试常问问题

#### Q1：为什么不是在事务方法最后直接发送 MQ？

事务方法最后一行执行时，数据库事务可能还没有真正提交。如果这时发送 MQ，消费者可能先消费消息并查询数据库，但数据还不可见。因此要使用 afterCommit，在事务真正提交后再发送消息。

#### Q2：afterCommit 执行失败怎么办？

afterCommit 中的动作失败不会回滚主事务。对于 MQ 发送失败，生产环境可以引入本地消息表或 Outbox 模式做可靠投递。对于 WebSocket 或工单失败，可以记录日志或通过补偿任务处理。

#### Q3：异步线程里事务还生效吗？

事务是线程绑定的，Spring 事务依赖 ThreadLocal。如果直接开启新线程执行数据库操作，原事务不会自动传播到新线程。

## 11. 亮点八：告警去重与幂等设计

### 11.1 为什么要做？

设备可能持续上报异常数据。

如果每次异常都新增一条告警，会导致告警记录爆炸、运维人员难以判断是否同一个问题、工单可能重复生成、WebSocket 可能重复推送大量相似告警、MQ 重复消费时数据异常。

因此项目必须做告警去重和幂等设计。

### 11.2 项目中如何实现告警去重？

告警检测时查询是否存在：

```text
同一设备
同一告警类型
同一告警指标
告警状态为未确认或已确认
```

如果存在，则更新当前值、阈值、告警文案、最近告警时间、告警次数和更新时间。

如果不存在，则新增一条告警记录。

### 11.3 项目中如何实现工单幂等？

自动工单生成时使用双保险：

1. 判断 `alarm_record.work_order_generated`；
2. 判断 `work_order` 中是否已有相同 `alarm_id`；
3. 数据库对 `work_order.alarm_id` 建唯一索引。

这样即使 MQ 重复消费，也不会重复生成工单。

### 11.4 带来的收益

1. 避免重复告警；
2. 避免重复派单；
3. 提高 MQ 重复消费容忍度；
4. 告警记录更清晰；
5. 运维处理更合理。

### 11.5 面试常问问题

#### Q1：MQ 重复消费会不会重复生成告警？

不会重复新增未恢复告警。项目通过设备、告警类型、告警指标、未恢复状态查询已有告警。如果存在，则更新告警次数和最近告警时间，而不是新增。

#### Q2：MQ 重复消费会不会重复生成工单？

不会。工单生成前会判断 `work_order_generated`，同时 `work_order.alarm_id` 有唯一索引。即使并发重复执行，数据库也能防止重复派单。

#### Q3：为什么只对未恢复告警去重？

因为已恢复告警代表一个历史问题已经结束。如果同一设备后续再次出现相同异常，应该生成新的告警记录，表示新的异常周期。

## 12. 项目涉及的八股知识点

### 12.1 RabbitMQ

#### 1. 为什么使用 MQ？

MQ 用于异步解耦、削峰填谷、失败重试和跨模块通信。本项目中用 RabbitMQ 解耦设备上报和告警检测。

#### 2. Exchange、Queue、RoutingKey 的关系是什么？

生产者发送消息到 Exchange，Exchange 根据 RoutingKey 和 Binding 规则将消息路由到 Queue，消费者从 Queue 中消费消息。

#### 3. 什么是消息可靠性？

包括生产者可靠投递、Broker 持久化、消费者确认、消费失败重试和死信队列兜底。当前项目实现了基础异步消费和重试，后续可扩展手动 ACK 和死信队列。

#### 4. 什么是重复消费？怎么解决？

重复消费是指同一条消息被消费者处理多次。解决方式是消费逻辑幂等。本项目通过告警去重和工单唯一索引保证重复消费不会产生严重副作用。

### 12.2 Redis

#### 1. Redis 为什么快？

Redis 基于内存操作，采用高效数据结构，单线程事件循环避免频繁上下文切换，同时使用 I/O 多路复用处理大量连接。

#### 2. 项目中用了哪些 Redis 数据结构？

| 数据结构 | 项目用途 |
|---|---|
| String | 设备心跳、设备最新状态、告警规则缓存 |
| Set | 在线设备集合、告警设备集合 |
| List | 连续异常滑动窗口 |

#### 3. 什么是缓存穿透？

查询不存在的数据，缓存和数据库都没有，导致请求每次都打到数据库。项目中对空规则列表也写入 Redis，避免规则为空时每条设备数据都查询数据库。

#### 4. 什么是缓存击穿？

热点 key 过期瞬间，大量请求同时打到数据库。后续可以通过互斥锁、逻辑过期等方式优化。

#### 5. 什么是缓存雪崩？

大量 key 同时过期，导致请求集中打到数据库。后续可以通过随机 TTL、分批过期等方式优化。

#### 6. Redis 和 MySQL 如何保证一致性？

本项目采用 Cache Aside：读时先 Redis，未命中 MySQL，再回写 Redis；写时先 MySQL，再删除 Redis。

### 12.3 WebSocket

#### 1. WebSocket 和 HTTP 有什么区别？

HTTP 是请求响应模型，服务端不能主动给客户端推送消息。WebSocket 是长连接，建立连接后服务端可以主动推送消息。

#### 2. WebSocket 适合什么场景？

适合实时告警、聊天、在线通知、监控大屏、实时状态更新。

#### 3. WebSocket 断开怎么办？

项目中在连接关闭和传输异常时移除 session。后续可以扩展心跳机制、断线重连、用户维度 session 管理。

### 12.4 Spring 事务

#### 1. `@Transactional` 什么时候会失效？

常见失效场景包括：方法不是 public、同类内部方法调用、异常被 catch 没有抛出、抛出的不是默认回滚异常、数据库引擎不支持事务、方法没有被 Spring 代理调用。

#### 2. 为什么事务和线程有关？

Spring 事务底层通过 ThreadLocal 绑定数据库连接。如果开启新线程，原事务上下文不会自动传过去。

#### 3. 为什么要 afterCommit？

因为某些后置动作必须依赖事务提交成功，例如发送 MQ、推送 WebSocket、创建工单。

### 12.5 MySQL

#### 1. 为什么使用逻辑删除？

逻辑删除可以保留历史数据，便于审计和恢复。项目中 `alarm_rule`、`work_order` 等表都使用 `deleted` 字段。

#### 2. 唯一索引有什么作用？

唯一索引用于保证数据唯一性。项目中 `alarm_rule.rule_code` 唯一，防止规则编码重复；`work_order.alarm_id` 唯一，防止重复派单。

#### 3. 为什么工单表要对 `alarm_id` 建唯一索引？

因为一个告警最多只能生成一个工单。即使业务代码判断失败或并发执行，数据库唯一索引也可以兜底。

### 12.6 幂等设计

#### 1. 什么是幂等？

同一个操作执行一次和执行多次，最终结果一致或可接受。

#### 2. 为什么 MQ 消费要幂等？

因为消息可能重复投递、消费者可能重试、网络异常可能导致重复处理。

#### 3. 项目中哪里体现了幂等？

告警记录去重、工单唯一索引、`work_order_generated` 标记、规则缓存删除后重建、WebSocket 推送失败不影响主流程。

### 12.7 分层架构

#### 1. 为什么 Controller 不直接调用 Mapper？

Controller 应该只负责参数接收和结果返回。业务逻辑应该放在 Service 层，Mapper 只负责数据库操作。这样职责清晰，后续维护和扩展更方便。

#### 2. DTO、VO、Entity 分别是什么？

| 类型 | 作用 |
|---|---|
| DTO | 接收请求参数 |
| Entity | 映射数据库表 |
| VO | 返回前端展示数据 |

#### 3. 为什么要统一返回 Result？

统一返回结构可以让前端处理更简单，也方便全局异常处理和错误码管理。

## 15. 面试讲项目时的推荐表达顺序

面试时不建议一开始就堆技术栈。

推荐表达顺序：

```text
1. 先讲业务背景：
   新能源充电桩运行过程中需要实时监测状态，异常时及时告警并派单处理。

2. 再讲系统闭环：
   设备上报数据 → 异步告警检测 → 实时推送 → 严重告警生成工单 → 运维人员处理。

3. 再讲核心技术：
   RabbitMQ 解耦上报和告警；
   Redis 缓存规则和维护连续异常窗口；
   WebSocket 实时推送；
   MySQL 唯一索引和业务字段保证幂等。

4. 最后讲你解决的问题：
   降低接口阻塞；
   降低规则查询压力；
   支持动态规则；
   支持连续异常判断；
   实现告警到工单的闭环。
```

可以概括成一句：

```text
这个项目的核心不是单纯做设备 CRUD，而是围绕设备运行数据构建了一套“异步检测、动态规则、实时推送、自动派单”的告警处理闭环。
```

## 17. 总结

本项目最终形成了比较完整的后端业务闭环：

```text
设备数据采集
→ Redis 状态缓存
→ RabbitMQ 异步告警
→ 动态规则检测
→ Redis 规则缓存
→ Redis 连续异常滑动窗口
→ 告警记录入库
→ WebSocket 实时推送
→ 严重告警自动生成工单
→ 工单接单、处理、完成或关闭
```

技术上覆盖了后端开发中非常常见、也非常适合面试展开的问题：

```text
异步解耦
缓存一致性
消息重复消费
幂等设计
事务提交时机
WebSocket 实时推送
Redis 数据结构应用
状态机流转
分层架构设计
```
