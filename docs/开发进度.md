### 3.6 基础阈值告警模块

MVP 阶段已实现三类阈值告警：

| 指标 | 条件 | 告警等级 | 告警类型 |
|---|---|---|---|
| temperature | temperature > 80 | 3 严重 | THRESHOLD |
| voltage | voltage < 180 | 2 重要 | THRESHOLD |
| networkDelay | networkDelay > 200 | 1 一般 | THRESHOLD |

告警去重维度：

```text
同一设备 + 同一告警类型 + 同一告警指标 + 未恢复状态
```

即：

```text
device_id + alarm_type + alarm_metric + alarm_status in (0, 1)
```

处理策略：

```text
不存在未恢复告警 → 新增 alarm_record
存在未恢复告警 → 更新 current_value、last_time、alarm_count、alarm_message
```

## 4.1 设备离线检测定时任务

当前状态：已完成，并已测试通过。

新增 / 修改内容：

```text
ChargeMonitorApplication.java       增加 @EnableScheduling
application.yml                     增加 charge-monitor.device.offline 配置
RedisKeyConstants.java              补齐设备状态、心跳、在线集合、告警集合 Key
DeviceOfflineProperties.java        新增设备离线检测配置类
DeviceOfflineCheckTask.java         新增设备离线检测定时任务
```

配置示例：

```yaml
charge-monitor:
  device:
    offline:
      enabled: true
      fixed-delay: 30000
      offline-seconds: 60
```

核心逻辑：

```text
定时查询所有未删除、未停用设备
→ 优先读取 Redis device:heartbeat:{deviceCode}
→ Redis 无数据时使用 MySQL device_info.last_heartbeat 兜底
→ 判断当前时间 - 最近心跳时间 是否超过 offlineSeconds
→ 超过则更新 device_info.online_status = 0
→ 从 Redis device:online:set 移除设备
→ 新增或更新 OFFLINE + heartbeat 离线告警
→ 将设备加入 Redis device:alarm:set
```

测试场景：

```text
CP-0001 ~ CP-0005 持续模拟 HTTP 上报，保持在线
CP-0007 只上报一次，之后停止上报
定时任务最终只发现 CP-0007 离线
```

测试结果：

```text
CP-0001 ~ CP-0005 持续在线，无误报
CP-0007 超过离线阈值后被标记为离线
CP-0007 从 device:online:set 移除
CP-0007 加入 device:alarm:set
alarm_record 生成 OFFLINE + heartbeat 告警
重复扫描不会重复新增离线告警，只更新 alarm_count 和 last_time
```

阶段价值：

```text
系统不只能识别“上报异常值”，也能识别“设备不再上报”这种真实运维异常。
```

```text
DeviceDataAlarmConsumer.java         新增消息消费者
DeviceDataServiceImpl.java           事务提交后投递 MQ，不再同步检测
```

核心流程：

```text
设备数据上报
→ 保存 device_data + 更新 device_info + 写 Redis
→ 事务提交成功后
→ DeviceDataAlarmProducer 投递消息到 RabbitMQ
→ DeviceDataAlarmConsumer 消费消息
→ 调用 AlarmDetectService.detectThresholdAlarms()
→ 新增或更新 alarm_record
```

关键设计点：

```text
1. 事务提交后才投递 MQ，避免消费者查不到数据
2. 消费者异常由 RabbitMQ 重试机制兜底
3. 上报接口不再同步返回告警结果，alarmTriggered 固定为 false
4. 告警结果以后以 alarm_record 查询为准
```

阶段价值：

```text
引入 RabbitMQ 构建设备数据异步告警链路，将设备数据上报与告警规则检测解耦，
降低上报接口阻塞风险，并为后续实时推送、自动工单和告警通知扩展提供事件驱动基础。
```

## 4.6 连续异常告警 CONTINUOUS（第五阶段核心）

当前状态：已完成，全部测试通过。

核心逻辑：

```text
每次设备数据上报
→ 判断本次是否命中规则（如 temperature > 75）
→ LPUSH 结果（1/0）到 Redis List
→ LTRIM 保留最近 window_size 条
→ 统计窗口中 1 的数量
→ 若 count >= trigger_count，触发 CONTINUOUS 告警
→ 告警恢复时自动清除 Redis 滑动窗口
```

Redis Key 设计：

```text
alarm:continuous:window:{deviceCode}:{ruleCode}
```

示例：`alarm:continuous:window:CP-0001:RULE_TEMP_CONTINUE`

关键设计点：

```text
1. Redis List + LPUSH + LTRIM 实现固定大小滑动窗口
2. 每次操作后 EXPIRE 续期 10 分钟，防止设备停报后 key 泄漏
3. 指标值为 null 时跳过（不 push），"没数据" ≠ "数据正常"
4. 窗口数据不足时（冷启动）不触发告警，避免误报
5. 告警恢复时主动删除窗口 key，防止残留数据导致立刻再次触发
6. 当前 CONTINUOUS 规则数量少，直接从 MySQL 查询，暂不引入缓存
```

测试场景：

```text
RULE_TEMP_CONTINUE：window_size=5, trigger_count=3, temperature > 75

连续上报 5 次 80℃ → 第 3 次触发告警 → alarmCount 累加至 3
恢复告警 → Redis 窗口被清除
恢复后再上报 2 次高温 → 未触发（窗口仅 2 个 1）
第 3 次高温 → 重新触发新告警
```

阶段价值：

```text
连续异常告警比单次阈值告警更贴近真实运维场景，
通过 Redis 滑动窗口实现"最近 N 次中有 M 次异常"的状态型异常判断，
避免偶发波动造成误报。
```

## 10. 当前面试表达素材

### 10.1 设备离线检测

```text
我在设备数据上报时维护 Redis 心跳 Key 和在线设备集合，同时通过 Spring Task 定时扫描设备台账，判断当前时间与最近心跳时间的差值是否超过离线阈值。若超过阈值，则更新设备在线状态、从 Redis 在线集合移除，并基于设备 ID、告警类型、告警指标和未恢复状态做告警去重，生成或更新 OFFLINE 离线告警，避免重复告警刷屏。
```

### 10.2 告警服务抽取

```text
为了避免设备数据上报服务职责过重，我将原本写在 DeviceDataServiceImpl 中的阈值告警判断、告警去重和告警记录生成逻辑抽取为 AlarmDetectService。拆分后，DeviceDataServiceImpl 只负责设备校验、数据入库和 Redis 状态更新，AlarmDetectService 专门负责告警检测。这样后续扩展动态规则表、RabbitMQ 异步告警和连续异常告警时，改动范围更清晰。
```
